# fadhilweblib Research Notes (April 5, 2026)

## Scope

Focused on official documentation and primary project sources that influence AOT transpilation, HMR, resumability/hydration, and native-core toolchain performance.

## Findings

1. **Vite HMR API constraints matter for static analysis**
   - Vite requires explicit `import.meta.hot.accept(` call shape and recommends guarding HMR sections so production builds can tree-shake them.
   - Implication: fadhilweblib compiler output should preserve analyzable HMR call patterns.
   - Source: https://vite.dev/guide/api-hmr

2. **Qwik resumability documents hydration-cost tradeoffs explicitly**
   - Qwik frames startup cost around listener restoration, component tree reconstruction, and state replay.
   - Implication: fadhilweblib's edge hydration planner should keep JS execution deferred and narrowly scoped.
   - Sources:
     - https://qwik.dev/docs/concepts/resumable/
     - https://qwik.dev/docs/advanced/qwikloader/

3. **Solid's fine-grained reactivity confirms targeted-update advantage**
   - Solid describes directly updating only affected reactive targets instead of broad component reruns.
   - Implication: compile-time extraction of reactive keys (e.g. `{{signal}}`) is aligned with low-overhead updates.
   - Source: https://docs.solidjs.com/advanced-concepts/fine-grained-reactivity

4. **Rolldown filter architecture quantifies Rust↔JS boundary overhead**
   - Rolldown documents plugin hook filters and reports measurable speedups by skipping unnecessary plugin calls in Rust before crossing into JS.
   - Implication: fadhilweblib should preserve coarse pre-filters before any JS transformation callbacks.
   - Sources:
     - https://rolldown.rs/apis/plugin-api/hook-filters
     - https://rolldown.rs/in-depth/why-plugin-hook-filter

5. **esbuild benchmark methodology highlights realistic baseline expectations**
   - esbuild publishes benchmark details and command reproducibility notes.
   - Implication: performance claims should include transparent command-level reproduction and percentile metrics.
   - Source: https://esbuild.github.io/faq/

- Keep transpiler scanner single-pass and linear; track scanner steps/char.
- Keep output runtime helpers at 0 bytes and emit direct native DOM API statements.
- Maintain explicit benchmark report with median + p95 and reproducible command.
- Add hardened variable naming mode for production output where source readability should be reduced.
